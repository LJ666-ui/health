## 四、AI使用规范（核心要求）

### 4.1 AI使用总原则

**AI是你的开发助手，不是你的替代品。你可以用AI辅助任何技术工作，但必须做到"用了能讲清楚"。**

核心规则只有一条：**AI帮你写的任何东西，你必须完全理解，答辩时能解释清楚每一行代码、每一个设计决策的原因。**

### 4.2 AI可以辅助的场景（几乎涵盖所有技术工作）

| 场景 | AI辅助方式 | 使用要求 |
|------|------------|----------|
| 系统架构设计 | AI辅助生成架构方案、提供设计思路 | 理解架构选择的原因，能解释为什么选这个方案 |
| 数据库表结构设计 | AI辅助生成ER图、表结构、索引设计 | 理解每个表/字段/索引的设计理由 |
| 微服务拆分策略 | AI辅助分析业务边界、建议拆分方案 | 能解释为什么这样拆、拆的依据是什么 |
| 编写Java/Python代码 | AI辅助生成业务代码、工具类、接口实现 | 逐行审查，理解每行代码逻辑 |
| SQL语句编写 | AI辅助生成复杂查询、存储过程 | 理解SQL执行逻辑，能手动优化 |
| 配置文件 | AI生成YAML/Properties/Dockerfile模板 | 根据项目实际修改参数，理解每个配置项含义 |
| Bug排查 | AI提供排查方向、分析错误日志 | 自己动手调试验证 |
| 数据清洗算法 | AI辅助实现去重/补全/归一化/异常检测代码 | 理解算法原理，能解释为什么选这个方法 |
| 机器学习模型 | AI辅助选择模型、编写训练代码、调参 | 理解模型原理，能解释超参数选择依据 |
| 神经网络设计 | AI辅助搭建LSTM/MLP网络结构 | 理解网络每一层的作用，能画出网络结构图 |
| NLP处理 | AI辅助实现分词/意图识别/实体提取 | 理解NLP流程，能解释规则设计思路 |
| Prompt工程 | AI辅助设计Prompt模板、优化提示词 | 理解Prompt设计原则，能解释为什么这样写 |
| ECharts配置 | AI生成图表配置代码 | 根据业务需求调整，理解配置项含义 |
| Docker部署脚本 | AI生成Dockerfile/Compose脚本 | 理解每个指令的作用 |
| 单元测试 | AI生成测试用例 | 自己补充边界用例 |
| 正则表达式 | AI生成复杂正则 | 理解并验证每条正则 |
| 文档润色 | AI辅助论文语言润色、格式调整 | 核心观点必须是自己想的 |

### 4.3 必须自己完成的核心事项（AI不可替代思考）

| 事项 | 原因 | 怎么做 |
|------|------|--------|
| 论文核心技术章节的**观点与结论** | 学术诚信，论文必须体现个人学术思考 | AI可辅助写代码和做实验，但分析结论必须自己写 |
| 系统创新点的**提炼与论证** | 创新点是论文灵魂，不可代劳 | 深入思考项目与同类系统的差异，自己总结创新点 |
| 实验结果的**分析与解读** | 必须体现个人对数据的理解 | AI可帮你跑实验，但数据解读和结论必须自己写 |
| 答辩时的**口头表达** | 必须自己能讲清楚 | 理解AI帮你写的所有内容，用自己的话讲出来 |
| 论文**整体逻辑与结构** | 论文是个人学术作品 | AI可润色段落，但章节逻辑和论证思路必须自己设计 |

### 4.4 AI使用实操建议

1. **先想后问**：先自己思考方案，再问AI获取更多思路，对比后选择最优方案
2. **分段使用**：不要一次性让AI生成整个模块，分功能点逐步生成+审查
3. **必做审查**：AI生成的代码逐行审查，确保符合项目规范，去除冗余
4. **理解透彻**：每段AI生成的代码，问自己"如果不用AI我能写出来吗？"，如果不能就深入学习
5. **记录过程**：保留AI交互记录，记录AI辅助了哪些工作，作为开发过程佐证
6. **论文标注**：论文致谢中说明使用了AI辅助编程工具（这是学术规范，不影响评分）

## 七、风险预案

| 风险 | 应对措施 |
|------|----------|
| 云服务器内存不足 | 精简部署服务、调整JVM参数、使用轻量级镜像 |
| 大模型推理慢 | 使用量化模型、优化Prompt、异步调用 |
| 微服务内存溢出 | 限制单服务内存、优化GC策略、排查内存泄漏 |
| 网络延迟大 | 本地开发调试为主、减少跨机器调用 |
| 数据库主从延迟 | 优化同步策略、关键写入走主库 |
| 开发进度滞后 | 砍掉非核心功能、优先保证主流程 |
